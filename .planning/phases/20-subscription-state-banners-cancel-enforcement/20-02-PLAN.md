Wave 1 backend — wire up the two server-side gaps that block the yellow banner from showing a real date and that prevent past_due tenants from being able to cancel.

Two surgical fixes in the canonical single-writer flow (no parallel writers introduced — see STATE.md Phase 19 P02 decision):

1. **cancelSubscription controller (`stripe.controller.ts` lines 457–539)**:
   - Branch on `tenantData.subscription?.status === 'past_due'` BEFORE calling Stripe.
   - past_due path → `stripe.subscriptions.cancel(stripeSubscriptionId)` (immediate). Single writer call: `subscriptionStatus: 'canceled'`, `plan: 'free'`, `stripeSubscriptionId: null`, `pastDueSince: null`, `cancelAtPeriodEnd: false`, `source: 'controller.cancelSubscription.past_due_immediate'`.
   - non-past_due path → existing `update({ cancel_at_period_end: true })` PLUS extract `cancel_at` from Stripe response and pass `cancelAt: new Date(updated.cancel_at * 1000)` to the existing `syncTenantPlanBillingSnapshot` call. This closes RESEARCH.md Open Question 3.

2. **stripeWebhook.ts `cancel_at_period_end` handler (lines 906–926)**:
   - The local `cancelAt = Timestamp.fromMillis(currentPeriodEndMs)` is already computed for `scheduledPlanAt`.
   - Add `cancelAt: cancelAt.toDate()` and `cancelAtPeriodEnd: true` to the existing `syncTenantPlanBillingSnapshot` call so the canonical `subscription.cancelAt` field gets populated. This closes Pitfall 2 from RESEARCH.md.
