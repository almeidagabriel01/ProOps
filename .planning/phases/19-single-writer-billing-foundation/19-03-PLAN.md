<objective>
Eliminate every parallel billing-state writer in the codebase by routing them through `syncTenantPlanBillingSnapshot` (extended in Plan 02). After this plan, the phase-gate grep `grep -rn "tenantRef\.\(set\|update\)" apps/functions/src/` returns ONLY (a) the function itself in `stripeWebhook.ts`, (b) the `whatsappEnabled` second write (Pitfall 2), (c) the explicitly-exempt trial writes documented in CONTEXT.md, and (d) the small addon-item-id-only direct write in `upsertTenantStripeBillingData` (whatsappOveragePriceId / whatsappOverageSubscriptionItemId — NOT in the locked `subscription.*` schema, marked with `// EXEMPT:` comment).

**Special case — applyScheduledPlanChanges.ts (Pitfall 5):** This cron has its own `db.runTransaction()`. Firebase Admin SDK does NOT support nested transactions, so we CANNOT call `syncTenantPlanBillingSnapshot` from inside it. Instead, we ADD `subscription.plan / subscription.scheduledPlan / subscription.scheduledPlanAt / subscription.scheduledPlanReason` writes directly inside the existing `tx.set(tenantRef, {...})` call.

**Critical user-doc / tenant-doc distinction:** Several functions in scope ALSO write to `users/{uid}` (e.g., `updateUserPlan`, `updateSubscriptionStatus`, `runStripeSync`, controller `userRef.set`). User-document writes are OUT OF SCOPE for Phase 19. Do NOT touch them. Only the `tenantRef.set` / `tenantRef.update` / `db.collection("tenants").doc(tenantId).set` calls that write billing-state fields are consolidated. The dotted-notation writes like `"subscription.cancelAtPeriodEnd": ...` on USER docs in `stripeHelpers.ts` are user-doc writes — leave them alone.

**Early-return exemption (billing-sync.service.ts ~line 71):** The branch that writes only `{ billingSyncedAt }` (no subscriptionStatus/plan/stripePriceId) is a sync timestamp, not billing state. Exempt it explicitly. Same for the cleanup writes that only set `{ billingSyncing: false }` and the `{ billingSyncing: true }` flag write — these are control-plane flags, not billing state.

**`upsertTenantStripeBillingData` is NOT a permitted exemption for billing state.** CONTEXT.md (locked Decisions) lists `subscription.stripeCustomerId` and `subscription.stripeSubscriptionId` inside the canonical `subscription.*` schema — they ARE billing state and MUST flow through `syncTenantPlanBillingSnapshot`. Only the addon-item identifiers (`whatsappOveragePriceId`, `whatsappOverageSubscriptionItemId`) — which are NOT in the `subscription.*` schema — may stay as a direct write, with an explicit `// EXEMPT: addon-item identifiers, not subscription state` comment. Task 2-I implements this split.
</objective>
