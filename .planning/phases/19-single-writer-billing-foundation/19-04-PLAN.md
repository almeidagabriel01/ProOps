Replace the two unbounded global `Map` caches with bounded `LRUCache` instances (lru-cache v11). One instance per file (NOT shared) — `billingStateCache` lives in `require-active-subscription.ts`, `PLAN_CACHE` lives in `tenant-plan-policy.ts`. Both: `max: 500`, `ttl: 30_000` **hard-coded** (no env override). Replace the Wave 0 LRU test scaffold with real eviction + TTL + clear assertions.

This plan has ZERO file overlap with Plans 02 and 03, so it runs in Wave 2 in parallel with Plan 02. It only depends on Plan 01 (the lru-cache install + test scaffold).

**Why separate instances:** Evictions in the billing-state cache (per-request enforcement) should not impact the plan-enforcement cache (per-write enforcement). They have different invalidation patterns: the billing-state cache is invalidated by every middleware miss; the plan cache is invalidated by `clearTenantPlanCache(tenantId)` calls from billing writes. Sharing one instance would conflate the two domains.

**TTL is HARD-CODED to 30_000 (CONTEXT.md locked decision):** Per the CONTEXT.md `LRU cache replacement` decision, both LRU instances use a fixed 30-second TTL with NO env-overridable knob. Plan 04 must NOT route the LRU `ttl` constructor option through `resolvePlanCacheTtlMs()` — that helper currently reads `TENANT_PLAN_CACHE_TTL_MS`, which would contradict the lock. Either delete `resolvePlanCacheTtlMs()` if it has no remaining callers OR leave it for non-LRU usage and simply do not call it for the constructor. Plan 04's job is to ensure the LRU constructor uses the literal `30_000` constant.

**Test helper compatibility (Pitfall 4):** `setTenantPlanCacheForTest` and `hasTenantPlanCacheForTest` in `tenant-plan-policy.ts` already use the Map's `.set` / `.get` API. LRUCache supports both — but `.get()` semantics differ slightly with TTL. Update the helpers to keep their existing return semantics. The `setTenantPlanCacheForTest` per-entry `ttlMs` override stays (test convenience only — production cache TTL remains hard-coded 30_000).
